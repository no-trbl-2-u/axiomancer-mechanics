# Spec 04 — Skills Engine

## Goal

Build the skill runtime on top of a 5-resource resonance economy. Replace the
placeholder `manaCost` field with a structured `ResourceCost` object covering
five resource types (Heart / Body / Mind / Fallacy / Paradox). Implement the
types, engine functions, and resolver wiring that generate resources from basic
actions, check resonance conditions, spend resources, and execute skill effects.

**Success state:** From the combat CLI, choosing the `skill` action lets the
player pick from their equipped skills, spends the skill's `resourceCost`,
generates the appropriate philosophical resource (Fallacy or Paradox), runs the
skill's effect resolution via the existing Spec 03 machinery, and logs the
outcome to the `combatEvents` stream. At least one damaging skill, one buff
skill, and one debuff skill are demonstrably working in the CLI. Resources are
tracked on `CombatState`, reset each combat, and correctly accumulate from
basic actions.

The skill library content (actual named skills, the early-game roster, and the
full e2e scripted test) lives in **Spec 04b**, which depends on this spec.

## Why now / dependencies

- **Unblocks:** Spec 04b (skill library + e2e), Spec 05 (equipment stats feed
  into skill scaling), Spec 06 (learnSkill + skill points), Spec 07 (enemy
  skill use), real combat variety.
- **Depends on:** Spec 02 (resolver pipeline — skills slot into it). Strongly
  benefits from Spec 01 and Spec 03 (skills reference the same effect and proc
  machinery).

## Current state

- `Skill` type exists with `manaCost: number` — this field will be removed and
  replaced with `resourceCost: ResourceCost`.
- `Character` has `mana` and `maxMana` — these will be removed.
- `CombatState` has no resource tracking — `combatResources: CombatResources`
  will be added and reset to zero in `initializeCombat`.
- `action: 'skill'` already exists in the action type but the resolver ignores
  it; `skillId` routing is missing.
- `Enemy.skills?` is typed but no engine consumes it.
- `Skills/skill.library.ts` was deleted in the audit; no skills exist.
- No engine functions exist in the Skills module yet.

## Resource Economy Design (locked 2026-05-11)

Five resource types, two tiers:

| Resource | Symbol | Generated by |
|---|---|---|
| Heart | ♥ | Heart stance basic actions |
| Body | ⚡ | Body stance basic actions |
| Mind | ★ | Mind stance basic actions |
| Fallacy | ⚖ | Using any Fallacy-category skill |
| Paradox | ∞ | Using any Paradox-category skill |

**Generation rules for basic actions:**

| Action | Outcome | Tokens generated |
|--------|---------|-----------------|
| attack | hit (player wins roll contest) | +3 of attacker's stance color |
| attack | miss (player loses roll contest) | +1 of attacker's stance color |
| defend | always | +5 of defender's stance color |

**Resonance** is a condition, not a currency. A skill requiring `{ body: 2, heart: 2 }`
implicitly requires Body-Heart Resonance — the player must hold ≥2 of each.
Spending the tokens dissolves the resonance. No separate resonance counter needed.

**Skill cost tiers (emergent from the cost object's shape):**
- Tier 1: Single stance color (e.g. `{ body: 3 }`). Fires early in a fight. Generates ⚖ or ∞ on use.
- Tier 2: Multi-color (e.g. `{ body: 2, heart: 2 }`). Requires mixed-stance play; resonance is implicit.
- Tier 3: Philosophical resource required (e.g. `{ mind: 2, fallacy: 1 }`). Needs a prior Tier 1+ skill this combat.

**Resources reset to zero at the start of each combat** (`initializeCombat`).
No cap on token accumulation for now; revisit after playtesting.

## Open questions

1. ~~**Mana economy.**~~ RESOLVED — replaced by 5-resource resonance system above.
   `Character.mana`, `Character.maxMana`, and `Skill.manaCost` are all removed.

2. ~~**Basic attack mana cost.**~~ RESOLVED — basic actions are free and *generate*
   resources per the table above. No resource cost for `attack` or `defend`.

3. **Skill damage formula.**
   > **(A)** `basePower + character.baseStats[scalingStat] × 0.5`. Predictable,
   easy to balance. Multiplier constant `SKILL_STAT_MULTIPLIER = 0.5` exported
   from `game-mechanics.constants.ts`.

4. **`scalingStat` source.**
   > **Per-skill.** Each skill declares its own `scalingStat`. The library will
   default it to `philosophicalAspect` but individual skills can deviate
   (e.g. a Heart skill that scales off Mind for a cross-stat design).

5. **`combatEffects` integration.**
   > **Same Spec 03 machinery.** Skill-applied effects route through
   `resolveEffectApplication`. The skill's `tier` determines the resist tier
   used (Tier 1 auto-applies, Tier 2 resisted, Tier 3 only nat 20 repels).

6. **Targeting.**
   > **`'self' | 'enemy'`** only. Multi-combatant (`'all_enemies'`, `'all_allies'`)
   is out of scope here — deferred to Spec 07.

7. **Skill slots.**
   > **Fixed at 4** (`equippedSkills` max length). Simple; Spec 06 can raise the
   cap via level-up if desired.

8. **Mid-combat swap.**
   > **No** — skill equipping only outside combat. Keeps in-combat decision
   space manageable.

9. **Learning gate.**
   > **Combat-stat-only for now.** Moral alignment gate (Spec 10) deferred.
   `SkillLearningRequirement` keeps `level`, `statRequirementType`,
   `statRequirementValue`, and `prerequisiteSkill`.

10. **Skill tier semantics.**
    > **`tier: 1 | 2 | 3`**, mirroring the effect tier system. The free-form
    `level` field is removed from the `Skill` type. Learning requirements can
    keep their own `level` field (character level gate).

11. **Library breadth.**
    > **Deferred to Spec 04b.** This spec ships the engine only. Spec 04b
    ships a minimum of 12 early-game skills across all stat × category × tier
    cells.

12. **Resource generation actor.**
    > **Player only.** `combatResources` lives on `CombatState`, not on the
    actor. Enemy skill use is Spec 07 — enemies keep their existing action
    logic for now.

13. **Resource cap.**
    > **No cap.** Token counts are unbounded integers for now. Revisit if
    hoarding or one-shot stacking becomes a balance issue in playtesting.

14. **Player action shape for skills.**
    > Extend `PlayerCombatAction` to add `skillId?: string`. When
    `action === 'skill'`, `skillId` is required; the resolver throws a
    descriptive error if it is absent.

## Proposed approach

1. **Add `ResourceCost` and `CombatResources` types** to `src/Skills/types.d.ts`:
   ```ts
   export interface ResourceCost {
     heart?: number; body?: number; mind?: number;
     fallacy?: number; paradox?: number;
   }
   export interface CombatResources {
     heart: number; body: number; mind: number;
     fallacy: number; paradox: number;
   }
   ```

2. **Update `Skill` type** in `src/Skills/types.d.ts`: remove `manaCost`,
   `level`, free-text `damageCalculation` and `effect`; add:
   - `tier: 1 | 2 | 3`
   - `resourceCost: ResourceCost`
   - `targetType: 'self' | 'enemy'`
   - `basePower: number`
   - `scalingStat: SkillsStatType`
   - `combatEffects?: SkillCombatEffects[]`

3. **Update `Character` type** in `src/Character/types.d.ts`: remove `mana` /
   `maxMana`; add `knownSkills: string[]` and `equippedSkills: string[]`.

4. **Add `combatResources: CombatResources`** to `CombatState` in
   `src/Combat/types.d.ts`; initialise all five counters to `0` in
   `initializeCombat` (`src/Combat/combat.reducer.ts`).

5. **Extend `PlayerCombatAction`** in `src/Combat/types.d.ts` with
   `skillId?: string`.

6. **Add `SKILL_STAT_MULTIPLIER = 0.5`** to `src/Game/game-mechanics.constants.ts`.

7. **Create `src/Skills/skill.engine.ts`** with these pure functions:
   - `generateBasicActionResources(resources, stance, outcome): CombatResources`
     where `outcome: 'hit' | 'miss' | 'defend'` — applies the generation table.
   - `generatePhilosophicalResource(resources, category): CombatResources`
     — adds 1 Fallacy or Paradox token based on skill category.
   - `canUseSkill(resources, skill): boolean` — iterates each key in
     `skill.resourceCost` and checks `resources[key] >= cost[key]`. Resonance
     is implicit: multi-key costs naturally require both types to be present.
   - `spendResources(resources, cost): CombatResources` — deducts cost; throws
     if any key would go negative (caller must guard with `canUseSkill`).
   - `calculateSkillDamage(character, skill, advantage): number` — formula:
     `skill.basePower + character.baseStats[skill.scalingStat] * SKILL_STAT_MULTIPLIER`,
     then applies the same advantage modifier used by basic attacks.
   - `executeSkill(state, skillId): RoundResolution` — validates equipped skill,
     calls `canUseSkill`, runs damage/effect resolution, calls `spendResources`,
     calls `generatePhilosophicalResource`, emits `RoundEvent`s.

8. **Wire `'skill'` into the resolver** (`src/Combat/combat.resolver.ts`):
   when `playerAction.action === 'skill'`, route through `executeSkill` instead
   of the attack/defend path. The resolver still emits the `combatEvents` stream
   in the same format.

9. **Update the combat CLI** (`src/CLI/combat.cli.ts`): extend
   `promptPlayerChoice` to show a Skills sub-prompt when the player selects
   `action: skill` — lists `equippedSkills` by name, player picks one, `skillId`
   is passed into `resolveCombatRound`.

10. **Update `characters.mock.ts`**: remove `mana`/`maxMana`; add
    `knownSkills: []` and `equippedSkills: []` defaults to `Player`.

11. **Unit tests** in `src/Skills/skill.engine.test.ts` for every function in
    `skill.engine.ts`. Integration test confirming `resolveCombatRound` with
    `action: 'skill'` emits correct events and returns updated `combatResources`.
